# DDTM — Decentralized Data Trading Marketplace

> 基于零知识证明与联盟链的数据要素交易系统原型  
> A blockchain-native data marketplace powered by ZKP and consortium chain

---

## 为什么做这个项目

全国已有 58 家数据交易机构，但普遍"有场无市"。核心问题不是缺交易所，而是**交易基础设施不够用**：

- **区块链仅做哈希存证**，不驱动交易本身——本质是"电子签章的区块链版"
- **隐私计算停留在概念验证**，远未成为交易标配
- **定价全靠双方谈**，缺乏自动化机制
- **DCB 数据资产凭证**是行政登记，不是可编程通证

DDTM 要做一个**链上驱动的数据交易引擎**——确权、定价、撮合、结算、交付全流程由智能合约和零知识证明驱动，区别于"中心化撮合 + 区块链存证"模式。

## 核心原理

### 1. 零知识证明驱动的数据交易

传统数据交易的核心矛盾：买方需要看数据才能判断价值，但卖方交出数据后买方就拿到了全部信息。DDTM 用三种 ZKP 电路解决：

#### π_key — 密钥一致性证明（1,322 约束，103ms）

证明卖方交付的解密密钥与挂牌时承诺的密钥一致，且加密绑定到买方公钥。

```
证明内容：
  MiMC(K_s) == h_Ks           —— 密钥与挂牌承诺一致
  K_s_enc = MiMC(pk_B, K_s, r) —— 密文绑定买方（第三方无法解密）

攻击防御：
  Wrong_Ks      → 证明不成立，自动拒绝
  Tampered_Ks   → 哈希不匹配
  Cross_pk_replay → 买方公钥绑定阻止重放
```

#### π_deliver — 数据交付证明（8,456 约束，327ms）

证明交付的加密数据与挂牌时承诺的数据一致，且质量达标。

```
证明内容：
  MiMC(K_s) == h_Ks           —— 密钥正确
  MiMC(D_blocks) == h_D       —— 数据未被篡改
  MiMC(C_blocks) == h_C       —— 密文与明文对应
  Q >= Q_req                   —— 质量达到买方阈值

攻击防御：
  D_good_C_bad   → 哈希链断裂，证明失败
  Low_Quality    → 质量不达标
  Wrong_Ks       → 密钥不匹配，数据无法解出
```

#### π_Q — 质量证明（2,310 约束预估，~51ms）

验证数据满足完整性、时效性、格式规范等多维质量指标。

```
证明内容：
  H(D || r_D) == c_D          —— 完整性（数据未被篡改）
  H(Q(D) || r_Q) == c_Q       —— 质量声明确实来自该数据
  Q_val >= Theta              —— 质量指标达标
  Q_val <= Ctx                —— 数据在时效范围内
```

### 2. Groth16 证明系统

选择 Groth16 而非 Plonk 的三个理由：

| 指标 | Groth16 | Plonk | 影响 |
|------|---------|-------|------|
| 验证 gas | ~200K | ~290K | 链上成本更低 |
| 证明体积 | ~200 bytes | ~400 bytes | 存储更省 |
| 验证时间 | **恒定**（1.7ms） | 随电路增长 | 关键优势 |

Groth16 的验证时间与电路规模无关——无论 331 约束还是 100 万约束，验证都是 O(1) 的。这对链上验证至关重要。

### 3. 链上状态机

10 个状态覆盖交易全生命周期，状态转换由合约方法强制执行：

```
LISTED ──(bid)──→ ESCROWED ──(submitProof)──→ QUALITY_VERIFIED
  │                   │                              │
  │                   │                     (startDelivery)
  │                   │                              │
  │                   │                          DELIVERING
  │                   │                        /          \
  │                   │                 (confirm)      (dispute)
  │                   │                      |              |
  │                   │                 CONFIRMED      DISPUTED
  │                   │                                   |
  │                   │                         (resolveArbitration)
  │                   │                              /          \
  │                   │                    CONFIRMED          REFUNDED
  │                   │
  └──(abort)──→ ABORTED
```

资金流规则：
- 卖方挂牌须质押 ≥ 售价 10% 的保证金
- 买方竞价时全额付款进入合约托管
- 成交后卖方收到售价+退还保证金，买方收到找零
- 卖方败诉：买方全额退款 + 卖方保证金被罚没

### 4. 联盟链 PBFT 共识

模拟 4/7/10 节点 PBFT 共识下的吞吐量表现：

| 节点数 | 并发 | TPS | P50 延迟 | P95 延迟 |
|:-----:|----:|----:|-------:|-------:|
| 4 | 10 | 49.4 | 5.4ms | 7.8ms |
| 4 | 500 | 307.8 | 3.2ms | 6.6ms |
| 7 | 500 | 245.4 | 3.9ms | 6.8ms |
| 10 | 500 | 246.1 | 5.1ms | 6.8ms |

瓶颈在 PBFT 三轮广播（Pre-Prepare + Prepare + Commit），而非 ZKP 验证。

## 快速开始

### 环境要求

| 组件 | 需要 |
|------|------|
| Node.js | ≥ 18（合约编译+测试） |
| Go | ≥ 1.21（ZKP 电路编译） |
| Python | ≥ 3.10（联盟链模拟） |

### 1. 编译并测试智能合约

```bash
cd ddtm_evm
npm install
npx hardhat compile
npx hardhat test
```

预期：7/7 测试通过，Gas 基准 ~395K/全流程，并发 TPS ~108。

### 2. 运行 ZKP 电路

```bash
cd ddtm_zkp

# π_key + π_deliver 全量攻击测试
./ddtm_v22 both

# 基准测试
./ddtm_zkp benchmark

# 编译 π_Q 电路（需要 Go）
go build -o pi_q_bin pi_q.go && ./pi_q_bin
```

预期：9/9 攻击测试通过，π_Q 约束 ~2,310。

### 3. 联盟链性能模拟

```bash
cd ddtm_zkp
python3 multi_node_sim.py
python3 experiment_report.py   # 生成 LaTeX 实验表格
```

预期：4/7/10 节点，49-308 TPS。

### 4. 一键全量实验

```bash
cd ddtm_zkp
bash run_all.sh
```
